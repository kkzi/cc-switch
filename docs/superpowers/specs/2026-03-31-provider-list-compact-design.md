# 供应商列表轻度紧凑化设计

## 背景

当前供应商列表界面在卡片内边距、卡片之间的垂直间距，以及搜索区/空状态等外围区块上都留有较多空间。视觉上较舒展，但在常规笔记本高度下，一屏可见的供应商数量偏少。

本次调整目标不是改风格，也不是做高密度信息面板，而是在保留现有信息层级和交互结构的前提下，做一次轻度紧凑化，让列表更利于扫描。

## 目标

- 让供应商列表整体更紧凑，但仍保持当前视觉语言
- 不减少展示信息
- 不改变主要交互逻辑
- 通过轻度收缩留白，让一屏多容纳一些内容

## 非目标

- 不重做供应商卡片的视觉风格
- 不改变字段顺序或隐藏任何信息
- 不调整功能行为、快捷键、故障转移逻辑、卡片操作逻辑
- 不做“高密度表格化”改造

## 设计选择

### 方案 1：只缩卡片之间的间距

优点：
- 改动最小
- 风险最低

缺点：
- 卡片本体仍然偏松
- 整体紧凑感提升有限

### 方案 2：只缩卡片内部间距

优点：
- 卡片本身更紧凑
- 用户能直接感知卡片高度变化

缺点：
- 外围区块和列表项之间的空白仍然明显
- 整体页面节奏不统一

### 方案 3：轻度整体压缩

优点：
- 页面从外到内都更紧凑
- 改动仍然可控
- 能兼顾列表可见数量与界面呼吸感

缺点：
- 涉及多个组件，需要做小范围回归

### 结论

采用方案 3：整体轻度压缩，但每一处只缩一档，不做激进收缩。

## 范围

受影响组件：

- `src/components/providers/ProviderList.tsx`
- `src/components/providers/ProviderCard.tsx`
- `src/components/providers/ProviderEmptyState.tsx`

测试关注：

- `tests/components/ProviderList.test.tsx`

## 详细设计

## 1. 列表外层间距

在 `ProviderList.tsx` 中轻度收缩以下区域：

- 页面根容器的纵向间距
- 搜索块与列表之间的间距
- 列表项容器的 `space-y`
- 无搜索结果提示块的内边距
- loading skeleton 的纵向间距与占位块高度

目标效果：

- 列表项之间更贴近，但仍保留卡片式节奏
- 搜索区域不会显得与列表割裂

## 2. 卡片本体间距

在 `ProviderCard.tsx` 中做轻度紧凑化：

- 卡片外层内边距从当前值下调一档
- 卡片主内容区 `gap` 缩小
- 标题/标签区域 `gap` 缩小
- 信息区垂直 `space` 缩小
- 操作区与主体区的横向空隙缩小

约束：

- 不改变文本大小
- 不改按钮显隐逻辑
- 不改变 hover、选中、故障转移等状态表现

目标效果：

- 卡片高度略降
- 标题、URL、状态和操作仍然易读

## 3. 空状态与无结果状态

在 `ProviderEmptyState.tsx` 和 `ProviderList.tsx` 中同步收缩：

- 空状态容器 padding
- 中央图标容器尺寸
- 标题与说明之间的距离
- 按钮区顶部间距
- 无结果提示块的 padding

目标效果：

- 页面在无数据时仍然显得整洁
- 不因紧凑化让空状态显得拥挤

## 4. 不改动的内容

以下内容明确保持不变：

- 文案内容
- 字号体系
- 图标风格
- 供应商卡片字段数量与顺序
- 供应商操作按钮集合
- 故障转移、OpenClaw、OMO 等分支逻辑

## 测试策略

### 组件回归

运行 `tests/components/ProviderList.test.tsx`，确保：

- loading skeleton 仍可渲染
- 空状态按钮仍可触发新增
- 搜索功能不受影响
- 卡片排序与行为回调不受影响

### 额外验证

运行受影响的供应商相关测试，确认列表与新增弹窗未出现意外回归。

## 风险

- 收缩过多会影响卡片可读性
  - 通过“只缩一档”控制风险
- 某些状态标签与操作区可能在窄屏下更拥挤
  - 保持字号和折行策略不变，仅缩留白
- loading / empty / no-result 状态若只改主列表，页面密度会不统一
  - 一并覆盖外围状态组件

## 实施步骤

1. 先为当前受影响组件建立回归基线
2. 调整 `ProviderList.tsx` 外围间距
3. 调整 `ProviderCard.tsx` 内部间距
4. 调整 `ProviderEmptyState.tsx` 与无结果提示间距
5. 运行相关组件测试和类型检查
